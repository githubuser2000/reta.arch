* 1. Welche Netzwerke sind dazugekommen?

Es ist *kein TCP/IP-Netzwerk* und kein externer Server dazugekommen.  
Es ist ein *internes Ausführungsnetzwerk* für Tasks, Queues, Prozessausführung, Kanäle, Resultate und deterministisches Gluing.

Die neue Datei ist:

#+begin_src text
reta_architecture/execution_network.py
#+end_src

Darin liegt die neue Runtime-Kapsel:

#+begin_src text
ExecutionNetworkCategory
SchedulerCategory
ChannelCategory
#+end_src

-----

* 2. Was ist im Execution-Netzwerk drin?

** 2.1 `ExecutionTask`

Ein Task ist eine lokale ausführbare Arbeitseinheit.

#+begin_src text
ExecutionTask
#+end_src

enthält:

#+begin_src text
index
payload
operation
priority
callable_path
metadata
#+end_src

Bedeutung:

#+begin_src text
Ein Morphismus oder eine reine Teilberechnung wird als ausführbarer Task verpackt.
#+end_src

Beispiel:

#+begin_src text
Chunk 17 einer Zeilenfilterung
Chunk 3 einer Primfaktorberechnung
Chunk 8 einer CSV-Dekodierung
Chunk 4 einer Spaltenprojektion
#+end_src

Wichtig ist =index=. Dadurch können Ergebnisse nach paralleler Ausführung wieder in Originalreihenfolge geklebt werden.

-----

** 2.2 `ExecutionResult`

Ein Ergebnis eines Tasks.

#+begin_src text
ExecutionResult
#+end_src

enthält:

#+begin_src text
task_index
value
operation
metadata
#+end_src

Bedeutung:

#+begin_src text
Worker-Prozess hat lokalen Task gerechnet.
Ergebnis wird noch nicht global als Wahrheit genommen.
#+end_src

Erst danach kommt:

#+begin_src text
deterministic_reduce
#+end_src

-----

** 2.3 `ExecutionRunResult`

Das ist der komplette Ausführungslauf.

#+begin_src text
ExecutionRunResult
#+end_src

enthält:

#+begin_src text
values
results
config
workers
task_count
queue_discipline
mode
#+end_src

Die wichtigste Eigenschaft darin ist:

#+begin_src text
parallel_or_serial_task_cover_glues_to_the_same_ordered_result
#+end_src

Also:

#+begin_src text
seriell gerechnet
=
prozessparallel gerechnet + deterministisch reduziert
#+end_src

* 3. Welche Queues, FIFO/LIFO/Stack/Priority sind dazugekommen?

In =execution_network.py= gibt es jetzt drei Scheduler-Formen.

-----

** 3.1 FIFO Queue

#+begin_src python
FifoTaskQueue
#+end_src

Operationen:

#+begin_src text
push
pop
#+end_src

Verhalten:

#+begin_src text
first in, first out
#+end_src

Das ist die Standarddisziplin.

Sinnvoll für:

#+begin_src text
Tabellenchunks
Zeilenchunks
CSV-Zeilen
Spaltenprojektionen
Breitenberechnung
Arithmetik-Batches
#+end_src

Beispiel:

#+begin_src text
Task 0 rein
Task 1 rein
Task 2 rein

raus:
Task 0
Task 1
Task 2
#+end_src

Das ist die sicherste Standardform, weil sie fair und leicht nachvollziehbar ist.

-----

** 3.2 LIFO Stack

#+begin_src python
LifoTaskStack
#+end_src

Operationen:

#+begin_src text
push
pop
#+end_src

Verhalten:

#+begin_src text
last in, first out
#+end_src

Sinnvoll für:

#+begin_src text
Stack-artige Verarbeitung
Tiefensuche
Nested Completion
rekursive Abhängigkeitsauflösung
Trace-Navigation
#+end_src

Beispiel:

#+begin_src text
Task 0 rein
Task 1 rein
Task 2 rein

raus:
Task 2
Task 1
Task 0
#+end_src

Wichtig: LIFO ist vorhanden, aber nicht automatisch überall verwendet. Für Tabellenchunks bleibt FIFO normalerweise besser.

-----

** 3.3 Priority Queue

#+begin_src python
PriorityTaskQueue
#+end_src

Operationen:

#+begin_src text
push
pop
#+end_src

Verhalten:

#+begin_src text
niedrigere priority zuerst,
bei gleicher priority nach index
#+end_src

Sinnvoll für:

#+begin_src text
Prompt-Aufgaben vor Batch-Aufgaben
kleine schnelle Jobs vor langen Jobs
Validierungsjobs mit hoher Priorität
sichtbare Ausgabe vor Audit-Nachbereitung
#+end_src

Beispiel:

#+begin_src text
priority 0: Prompt-Antwort
priority 1: sichtbare Tabelle
priority 2: Audit
priority 3: Cache-Nachbereitung
#+end_src

Diese Queue ist jetzt als Architekturmechanik vorhanden, aber die konkrete Runtime nutzt aktuell vor allem FIFO für Prozesschunks.

* 4. Welche Datenströme sind dazugekommen?

Es gibt jetzt mehrere interne Datenströme.

** 4.1 Task-Strom

#+begin_src text
ExecutionTask
        ↓
ExecutionQueue / FifoTaskQueue / LifoTaskStack / PriorityTaskQueue
        ↓
dispatch_task
        ↓
ExecutionResult
#+end_src

Dieser Strom beschreibt:

#+begin_src text
lokale Arbeitseinheiten werden geordnet,
ausgeführt,
als Ergebnisse gesammelt.
#+end_src

-----

** 4.2 Resultat-Strom

#+begin_src text
ExecutionResult
        ↓
deterministic_reduce
        ↓
OrderedResultSection
#+end_src

Das ist der wichtigste Strom.

Er verhindert, dass parallele Prozessfertigstellung die Ausgabe-Reihenfolge verändert.

Also nicht:

#+begin_src text
wer zuerst fertig ist, kommt zuerst in die Tabelle
#+end_src

sondern:

#+begin_src text
nach task_index / Originalindex sortieren
dann kleben
#+end_src

-----

** 4.3 Prozess-Chunk-Strom

In:

#+begin_src text
reta_architecture/parallel_execution.py
#+end_src

werden Chunks erzeugt:

#+begin_src text
Daten
        ↓
_chunks(...)
        ↓
ExecutionTask pro Chunk
        ↓
Prozesspool
        ↓
ExecutionResult
        ↓
deterministische Reduktion
#+end_src

Der zentrale interne Helfer ist:

#+begin_src python
_pool_map_ordered(...)
#+end_src

Der nutzt jetzt, wenn möglich:

#+begin_src text
ExecutionNetworkConfig
ExecutionTask
execute_tasks_deterministically
#+end_src

Also:

#+begin_src text
parallel_execution.py benutzt execution_network.py
#+end_src

-----

** 4.4 Persistenz-Batch-Strom

In:

#+begin_src text
reta_architecture/persistence.py
#+end_src

gibt es Batch-Vorbereitung:

#+begin_src text
viele Sektionen / Snapshots / Cache-Werte
        ↓
prozessparallel JSON + SHA-256 Digest vorbereiten
        ↓
seriell in SQLite schreiben
#+end_src

Wichtig:

#+begin_src text
Die Vorbereitung kann parallel sein.
Die SQLite-Schreiboperation bleibt seriell.
#+end_src

Das ist richtig, weil parallele DB-Schreibzugriffe sonst leicht Locking-Probleme erzeugen.

-----

** 4.5 Audit-Strom

#+begin_src text
ExecutionRunResult / Validation / Cache / Persistence
        ↓
record_audit_event
        ↓
audit_events
#+end_src

Das ist der Nachweisstrom:

#+begin_src text
Was wurde ausgeführt?
Was wurde validiert?
Was wurde persistiert?
Was wurde gecached?
#+end_src

* 6. Welche Semaphoren sind dazugekommen?

Es gibt:

#+begin_src python
ResourceSemaphore
#+end_src

und im Bundle:

#+begin_src text
cpu_semaphore
file_io_semaphore
output_semaphore
#+end_src

Konkret in:

#+begin_src python
ExecutionNetworkBundle
#+end_src

-----

** 6.1 CPU-Semaphor

#+begin_src text
cpu_semaphore = ResourceSemaphore(config.max_workers)
#+end_src

Sinn:

#+begin_src text
CPU-Arbeit auf konfigurierte Workerzahl begrenzen.
#+end_src

-----

** 6.2 File-IO-Semaphor

#+begin_src text
file_io_semaphore = ResourceSemaphore(max(1, min(4, config.max_workers)))
#+end_src

Sinn:

#+begin_src text
Dateizugriffe nicht unbegrenzt parallelisieren.
#+end_src

Gerade bei Manifest-Hashing oder CSV-Zugriffen kann zu viel paralleles IO langsamer werden.

-----

** 6.3 Output-Semaphor

#+begin_src text
output_semaphore = ResourceSemaphore(1)
#+end_src

Sinn:

#+begin_src text
sichtbare Ausgabe nicht vermischen.
#+end_src

Also:

#+begin_src text
nur ein Ausgabepfad gleichzeitig
#+end_src

-----

** Ehrlicher technischer Hinweis zu den Semaphoren

=ResourceSemaphore= verwendet aktuell intern:

#+begin_src python
threading.BoundedSemaphore
#+end_src

Das ist ein lokaler In-Process-Semaphor, kein echter geteilter =multiprocessing.Semaphore= zwischen mehreren Prozessen.

Die tatsächliche Prozessanzahl wird durch:

#+begin_src text
multiprocessing.Pool(processes=workers)
#+end_src

begrenzt.

Das heißt:

#+begin_src text
Prozessparallelisierung: ja
gemeinsam zwischen Prozessen geteilter Semaphor: nein, aktuell nicht
#+end_src

Für echte prozessübergreifende Semaphoren müsste man später gezielt auf:

#+begin_src python
multiprocessing.BoundedSemaphore
#+end_src

oder Manager-/Shared-State-Mechanik gehen. Aktuell ist das nicht nötig, weil SQLite-Writes seriell bleiben und Prozesspools ohnehin begrenzt sind.

#+begin_src text

#+begin_src text
